# Learning: 2026-04-23

## Context

Build audit after evals 51–55 (dh-sandbox repos). All 5 configs passed `qlty plugins list` locally, but 4 repos had "Build errored" in Qlty Cloud — all caused by runtime plugin failures, not TOML syntax errors. Root causes discovered:

- `tsc`: marked `hidden = true` in the registry with no `known_good` version — fails config validation before running, but this wasn't caught before pushing the PR
- `eslint` with legacy `.eslintrc.js`: Qlty's eslint plugin uses ESLint 9.7.0 (flat config only) — legacy config silently ignored at runtime, generating `eslint --config ""` which exits with code 2
- `vale`: requires `.vale/styles` directory populated by `vale sync` at runtime — fails immediately if styles not committed

The skill currently ends after writing the TOML and opening a PR. There is no local verification step. All runtime failures surfaced only after Qlty Cloud ran the build.

## What we learned

1. **Config syntax validation is insufficient — runtime crashes exist**: A config can be structurally valid TOML, pass `qlty plugins list`, and still cause "Build errored" in Qlty Cloud when a plugin crashes at runtime. The skill needs a local smoke-test before opening the PR.

2. **`qlty check --sample` catches runtime crashes locally**: Running `qlty check --sample` executes all enabled plugins on a small random sample of real files. It surfaces exit-code errors, missing binaries, and misconfigured plugins within seconds — the same failures that would appear as "Build errored" in Qlty Cloud, but before the PR is open.

3. **When a plugin fails: check docs → fix → or comment out**: The right failure-handling sequence is:
   - Check the debug file Qlty prints and https://qlty.sh/d/lint-error
   - Attempt a fix (config path, extra_packages, plugin version)
   - If not fixable: comment out the entire `[[plugin]]` block with a one-line note above it explaining why (e.g., `# disabled: eslint flat config incompatible with project's legacy xo setup`)
   - Run `qlty check --sample` again to confirm clean pass before opening the PR

4. **Do not open the PR while any plugin is erroring**: An erroring plugin causes "Build errored" in Qlty Cloud for the entire build — not just that plugin. All PR feedback is blocked until the error is resolved. A clean local `--sample` run is a prerequisite for the PR.

5. **Comment-out format**: When disabling a broken plugin, keep the block as commented TOML lines so it's easy to re-enable later. The explanatory comment goes on the line immediately above `# [[plugin]]`:

   ```toml
   # disabled: ESLint 9 flat config required but project uses legacy .eslintrc.js
   # [[plugin]]
   # name = "eslint"
   # mode = "comment"
   # config_files = [".eslintrc.js"]
   ```
