# Learning: 2026-04-16 (batch d — evals 41-50)

## Context

10 new eval runs across simple_form (Ruby/haml-lint), qlty-test-ts-knip (TypeScript/knip), Slim (PHP/php-codesniffer), click (Python/mypy strict), qlty-test-prisma (TypeScript/prisma), dry-core (Ruby/standardrb), qlty-test-dotenv (dotenv-linter), qlty-test-oxc (TypeScript/oxc+eslint), chalk (Node.js/editorconfig-checker), puma (Ruby/config refinement). 9/10 passed without format errors on first attempt. 1 "Build errored" in Slim due to `psalm` not being in the public plugin registry.

## What we learned

1. **`psalm` is NOT in the public Qlty plugin registry — causes Build errored**: One agent found `psalm.xml` in the Slim PHP repo and added a `[[plugin]] name = "psalm"` block. The `psalm` plugin is not available in the public `[[source]] default = true` registry (same pattern as `radarlint-java` in batch c). The agent correctly used `php-codesniffer` and `phpstan`, but adding `psalm` caused "Build errored". **Fix:** Add `psalm` to the SKILL.md plugin table with a "(not in public registry — omit)" note in Decision 1. For PHP projects, the supported static analysis tools are `phpstan` only; `psalm` is not supported by Qlty Cloud.

2. **CRITICAL callout block is working — 0 format errors this batch**: The prominent CRITICAL callout added in Phase 4 after batch c (which warned about `[cli]`/`[settings]` sections, old source formats, missing `[[source]]`, etc.) resulted in zero format errors across all 10 agents. This is a significant improvement over batch c where 5/10 agents had format errors. The callout approach is confirmed effective.

3. **`standardrb` vs `rubocop` detection works reliably**: Two repos in this batch had `.standard.yml` (dry-core and sidekiq from batch c). Both agents correctly chose `standardrb` over `rubocop` and did not enable both. The mutual-exclusivity guidance is working.

4. **`knip` coexists with `eslint` correctly**: The knip eval confirmed that agents understand knip detects dead code/unused exports (set to `monitor`), while eslint handles linting — different tools, both should be enabled. No agent tried to replace eslint with knip.

5. **`oxc` and `eslint` coexist correctly (unlike biome)**: When both `.oxlintrc.json` and `eslint.config.mjs` were present, the agent correctly enabled both plugins. The skill guidance that "oxc does NOT replace eslint (unlike biome)" is understood. This is the opposite of the biome-exclusive rule.

6. **`dotenv-linter` + `[[exclude]]` for example files**: The dotenv eval confirmed the `[[exclude]]` syntax for scoping a plugin to skip certain files works correctly. Agents use `[[exclude]] plugins = [...] file_patterns = [...]` syntax properly.

7. **`editorconfig-checker` requires `[[exclude]]` for generated directories**: When editorconfig-checker is enabled, agents correctly add `[[exclude]]` blocks or `exclude_patterns` to skip `node_modules/`, `dist/`, and vendored directories. Without this, editorconfig-checker would flag auto-generated/vendored files.

8. **`affects_cache` on mypy correctly invalidates cache when `pyproject.toml` changes**: The click eval confirmed agents understand `affects_cache = ["pyproject.toml"]` is needed for mypy when its config lives in pyproject.toml. Same pattern applies to ruff. This prevents stale analysis results when config changes.

9. **Config refinement (eval 50 — puma)**: Agents can correctly find an existing qlty.toml, read it, and add incremental improvements (reek, triage rules, smells thresholds) without rewriting from scratch. The instruction "do not start from scratch" was followed.

10. **`psalm` discovery: add to unsupported PHP plugins list**: Following the pattern of `radarlint-*`, `psalm` should be added to a "not in public registry" list in SKILL.md for PHP projects. The supported PHP static analysis stack is: `php-codesniffer` (or `php-cs-fixer`), `phpstan` — that's it.
