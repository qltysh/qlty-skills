# Learning: 2026-04-29

## Context

Evals 51–55: setup-qlty-toml run end-to-end on 5 real repos (fastapi, express, axios, mux, ruff). 4/5 passed Qlty Cloud builds on first PR push. Axios errored in the cloud despite a clean local `qlty check --sample`.

## What we learned

1. **Always use CJS syntax in `.qlty/configs/eslint.config.js` — `.mjs` does not work.**
   Qlty copies config files to its cache renaming them to `.js` regardless of the source extension. An `eslint.config.mjs` with ESM `import` syntax was renamed to `eslint.config.js` in the cache and failed with `SyntaxError: Cannot use import statement outside a module`. The previous guidance to use `.mjs` for ESM projects was incorrect — always write CJS (`const x = require(...); module.exports = [...]`) regardless of the project's module type. This was confirmed by busting the eslint cache and re-running: the local sample failed once the old cached config was cleared.

2. **`@eslint/js` version must match Qlty's ESLint 9.7.0 — do NOT use `@eslint/js@10.x`.**
   The axios eval used `extra_packages = ["@eslint/js@10.0.1", ...]`. This is the version for ESLint 10 and is incompatible with Qlty's bundled ESLint 9.7.0. The cloud build errored, but the local `qlty check --sample` passed — the cached eslint.config.js from a prior run masked the incompatibility locally. Always use `@eslint/js@9.7.0` or `@eslint/js@^9.0.0` as the extra_package version.

3. **`qlty check --sample` does not catch all cloud-specific failures — package version conflicts are one gap.**
   When `extra_packages` specifies a version incompatible with Qlty's bundled tool (e.g. `@eslint/js@10.x` alongside ESLint 9.7.0), npm may resolve a compatible fallback locally while the cloud uses a fresh, stricter install environment. Result: local sample passes, cloud build errors. Add a validation step: after writing `extra_packages`, cross-check each package's major version against the plugin's bundled tool version.

4. **`eslint-plugin-react` requires v7.33+ for flat config (`react.configs.flat.recommended`)**
   The ruff eval used `eslint-plugin-react@7.31.11` which lacks the `.flat` API introduced in v7.33.0. When the flat config references `react.configs.flat.recommended`, ESLint throws `TypeError: Cannot read properties of undefined (reading 'recommended')`. Fix: pin to `eslint-plugin-react@7.37.5` (latest as of 2026-04-29) or any version ≥ 7.33.0.

5. **`rustfmt` with `style_edition = "2024"` in `rustfmt.toml` fails under Qlty's bundled rustfmt 1.77.2.**
   Qlty bundles rustfmt 1.77.2 (stable 2024-03-21). The `style_edition = "2024"` option is only supported in nightly rustfmt and a newer stable. If a repo's `rustfmt.toml` has this option, rustfmt exits with error. Fix: add the Rust source dirs to `exclude_patterns` (if formatting is enforced by CI separately), or comment out the `rustfmt` plugin block with a note.

6. **Test fixture directories with intentional syntax errors must be explicitly excluded.**
   The ruff eval's `crates/ty_completion_eval/truth/` directory contains Python files with deliberate syntax errors used as type-checker test fixtures. Ruff fails on these with parse errors. When you see a `crates/**/truth/` or similar fixture directory in a Rust/Python polyglot repo, add `crates/**/truth/**` (or the specific path) to `exclude_patterns`. More generally: always inspect intentional-error fixture directories during Phase 1 and exclude them up front.

7. **`editorconfig-checker` may not be present in the local Qlty registry version — check before including.**
   The mux eval found `editorconfig-checker` was not in the locally installed qlty registry (`qlty plugins list` returned nothing for it), even though the repo has `.editorconfig`. This caused the plugin block to silently fail. Always verify a plugin exists via `qlty plugins list` locally (or check the live GitHub registry) before adding it.
