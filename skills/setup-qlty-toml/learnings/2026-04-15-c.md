# Learning: 2026-04-15 (batch c — evals 31-40)

## Context

10 new eval runs across devise (Ruby/Rails), hono (TypeScript/biome), sidekiq (Ruby), flask (Python/semgrep), flysystem (PHP/php-cs-fixer), gin (Go/golangci-lint), commons-lang (Java/pmd), qlty-test-ts-multi-eslint (TypeScript/multi-eslint), serde (Rust/workspace), and mkdocs (Python/vale). All 10 passed after manual correction of 5 agent-generated configs that caused "Build errored".

## What we learned

1. **`[cli]` and `[settings]` sections from `qlty init` output must NOT appear in qlty.toml**: One agent ran `qlty init` and included the resulting `[cli]` (with `version = "..."`) and `[settings]` (with `root_dir = "."`) sections in the committed qlty.toml. These are internal qlty CLI metadata sections — not valid qlty.toml config fields. Including them causes Build errored. **Fix:** Added explicit warning in Phase 4: "Never include `[cli]` or `[settings]` sections — these appear in `qlty init` output but are not valid qlty.toml config fields."

2. **Old `[[sources.community]]` format is deprecated — causes Build errored**: One agent used an old source block format: `[[sources.community]] name = "qlty" repository = "https://..." tag = "v0.1.156"`. The current format is `[[source]] name = "default" default = true`. The old format may appear in cached web examples and training data. **Fix:** Added the deprecated format explicitly to the Phase 4 template warning alongside the correct form.

3. **`[[source]]` block omitted by multiple agents despite "Required" comment**: 3 of the 5 failing agents either omitted the `[[source]]` block entirely or generated it with wrong fields. The "Required" comment in the template is not strong enough. **Fix:** Elevated the `[[source]]` requirement into a bold callout box directly before the Phase 4 template, listing common wrong forms to avoid.

4. **`[[source]] directory = "https://..."` is wrong — `directory` is for local paths only**: One agent used `[[source]] name = "default" directory = "https://github.com/qltysh/plugins"`. The `directory` field on a `[[source]]` block is for local filesystem paths (e.g., a private plugin registry on disk). For the default Qlty cloud registry, use `default = true` — no URL needed. **Fix:** Added note to the `[[source]]` guidance: "Do NOT set `directory` to a URL — use `default = true` for the standard Qlty registry."

5. **Old `[plugins.releases]` / `[plugins.enabled]` / `[[plugins.enabled]]` format causes Build errored**: One agent generated the deprecated qlty.toml v0 format that uses `[plugins.releases]` for version pins and `[[plugins.enabled]]` for plugin blocks. The current format uses `[[plugin]]` blocks with inline `mode`, `version`, `config_files`, etc. **Fix:** Added a "Deprecated formats to avoid" callout in Phase 4.

6. **`[exclude_patterns] patterns = [...]` is wrong nesting — causes Build errored**: One agent used `[exclude_patterns]` as a TOML section header with a `patterns = [...]` sub-key. The correct form is a top-level array: `exclude_patterns = ["..."]`. **Fix:** Reinforced in the Phase 4 template comment and added to the deprecated-formats callout.

7. **`min_duplication_lines` is not a valid smells field**: One agent used `[smells] min_duplication_lines = 30`. This field does not exist. The correct way to control duplication sensitivity is `[smells.duplication] nodes_threshold = N` (higher = less sensitive). **Fix:** Added to Decision 6 smells table as an explicit "invalid field" note, alongside the correct `nodes_threshold` field.

8. **`radarlint-java` (and `radarlint-*`) are NOT in the public Qlty plugin registry**: Including `radarlint-java` in the config causes Build errored because the plugin is not resolvable from the public `[[source]] default = true` registry. It requires enterprise/commercial Qlty access. **Fix:** Added "(enterprise — requires separate setup)" note next to all `radarlint-*` mentions in Phase 1 and Decision 1. Agents should omit radarlint plugins unless the user explicitly confirms they have enterprise access.

9. **Sidekiq: rubocop linting-only scenario occurs naturally when repo uses StandardRB**: sidekiq uses `.standard.yml` (StandardRB as formatter), so the linting-only [[triage]] pattern was applied in practice. The skill's rubocop-linting-only guidance worked correctly when the team's formatter was identified. Confirmed the [[triage]] Style/Layout → monitor pattern is the right approach in this scenario.
