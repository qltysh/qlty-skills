# Learning: 2026-04-15 (batch b — evals 26-30)

## Context

5 new eval runs covering advanced config features: skip_upstream + .qlty/configs/ (qlty-test-tsc/TypeScript), version pin + [[triage]] set.level + [[exclude]] (Alamofire/Swift), extra_packages for PHP/Composer + [[exclude]] (pest/PHP), [[triage]] set.category + set.ignored + [[exclude]] (openapi-backend/TypeScript), smells.duplication.nodes_threshold + filter_patterns + language.ruby.smells (qlty-test-smells/Ruby+Python).

## What we learned

1. **`tsc` plugin causes build errors on complex real-world TypeScript projects**: Both `got` and `openapi-backend` (complex open-source TypeScript repos) showed "Build errored" with `tsc` enabled — even though their v1 configs (from evals 1-20) also failed. Simple/controlled test repos (qlty-test-tsc, qlty-test-mono) work fine with `tsc`. **Fix:** Add guidance to the skill: "Only enable `tsc` if the project already passes `tsc --noEmit` in CI. Skip `tsc` on complex repos where compilation status is unknown." Also note that `skip_upstream = true` reduces scope but does not prevent errors from broken upstream type contexts.

2. **xo-based TypeScript projects should NOT use the eslint plugin**: `got` uses `xo` (an eslint wrapper CLI) not raw eslint. Trying to configure raw eslint with `@typescript-eslint` packages failed because `xo` bundles its own eslint internals. **Fix:** Added to the skill's Phase 1 config table: check for `xo` in devDependencies — if present, skip the eslint plugin (xo is a CLI linter, not an eslint config that Qlty can run directly).

3. **ESM projects require `.eslintrc.cjs` not `.eslintrc.js` for config stored in `.qlty/configs/`**: Projects with `"type": "module"` in package.json treat `.js` files as ESM. A `.eslintrc.js` using `module.exports = {...}` will fail because that's CommonJS syntax. Use `.eslintrc.cjs` (explicit CommonJS file extension) for ESM projects. **Fix:** Added warning to the `.qlty/configs/` guidance in Decision 3.

4. **`@typescript-eslint` scoped packages only exist up to v6.x**: The agent generated `@typescript-eslint/parser@7.18.0` and `@typescript-eslint/eslint-plugin@7.18.0` — these don't exist. In v7+, the package was rebranded to the unscoped `typescript-eslint` (no `@` scope). Always use v5.62.0 or v6.x for `@typescript-eslint/parser` and `@typescript-eslint/eslint-plugin`. **Fix:** Added explicit note to Decision 3 extra_packages guidance: cap `@typescript-eslint/*` at v6.x; v7+ is a different package name.

5. **`set.category` in `[[triage]]` may not be supported by Qlty Cloud**: Adding `set.category = "security"` to triage blocks caused "Build errored" on openapi-backend. Removing it resolved the error. The field appears in documentation but is not reliably supported. **Fix:** Marked `set.category` as "use with caution — may cause build errors" in the triage guidance in Decision 8 and the triage reference file.

6. **`set.ignored = true` in `[[triage]]` may not be supported**: Using `set.ignored = true` also coincided with build errors. Replacing with `set.mode = "monitor"` was safer. **Fix:** Changed the [[triage]] guidance in Decision 8 to prefer `set.mode = "monitor"` or `set.mode = "disabled"` for suppression; note that `set.ignored` behavior may not be consistent across Qlty versions.

7. **PHP `package_filters` works for unscoped composer packages**: `package_filters = ["mrpunyapal"]` correctly matched `mrpunyapal/peststan` from composer.json. Unlike npm's `@scope/package` naming, Composer packages are `vendor/name` — prefix-matching on `vendor` works correctly. **Fix:** Added clarification to the `package_filters` warning: the limitation applies specifically to npm scoped packages (`@scope/*`); Composer and RubyGem package filters work correctly as long as the filter matches the package name prefix.
